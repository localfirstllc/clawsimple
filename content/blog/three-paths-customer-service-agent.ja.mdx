「Agent ランタイムに Skill ファイルを追加するだけでカスタマーサービス Agent が作れる」というアイデアを初めて聞いた人の多くは、_そんなに簡単なはずがない。本物のソリューションのはずがない。_ と感じます。そして本能は、Coze のようなビジュアルワークフロービルダーを選ぶか、LangGraph で Agent ループをゼロから書き始める方向へ向かいます。

現実はその逆です。既存の Agent ランタイムに Skill を追加することは妥協ではありません。インフラを再発明せずにカスタマーサービス Agent を完全に制御したい人にとって、**最も実用的な道**なのです。

## まず、レイヤーを理解する

アプローチを比較する前に、各要素がスタックのどこに位置するかを理解する必要があります：

```
┌─────────────────────────────────────────────┐
│            顧客が見る部分                      │
│   WeChat · Feishu Bot · Web Widget · ...     │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│       Agent プラットフォーム（Hermes / OpenClaw）│
│  チャネル · ツールループ · セッション · MCP · Cron│
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│             Skill / プロンプトレイヤー          │
│   分類 · ナレッジベース検索 · チケット生成        │
└─────────────────────────────────────────────┘
```

**Hermes と OpenClaw はどちらもフルスタックの Agent プラットフォームです——それぞれがメッセージチャネル（Telegram、WeChat、Feishu など）と Agent ランタイム（ツールループ、セッション、記憶、MCP、Cron）を兼ね備えています。両者は代替関係であり、階層関係ではありません。どちらか一つを選びます。**

あなたの Skill は最下層にあります。それは「カスタマーサービスをどう行うか」を定義します。その上のすべて——Agent ランタイムとメッセージチャネル——は、選んだプラットフォームにすでに組み込まれています。ツール呼び出しループを書く必要も、セッションマネージャーを書く必要も、メッセージアダプターを書く必要もありません。それらは既に存在しています。

つまり「Hermes にカスタマーサービス Skill を追加する」ことは半端な対策ではありません。それ**こそが**カスタマーサービス Agent を構築する完全な道なのです。Coze との違いはインターフェースだけです：

- **Coze**：キャンバス上でノードをドラッグし、プラグインにチェックを入れ、ナレッジベースをアップロードします。
- **Hermes + Skill**：`SKILL.md` を書き、ナレッジベースを設定し、ツールを接続し、`hermes` を実行します。

成果物は同じです：メッセージチャネルを通じて顧客の質問を受け取り、複数ターンの会話を行い、ナレッジベースを検索し、ツールを呼び出し、チケットの下書きを生成するシステムです。

## 3つの道を比較する

| | Coze / Dify | Hermes + Skill | スクラッチ開発（LangGraph等） |
|---|---|---|---|
| **Agent ランタイム** | プラットフォームホスト | Hermes が提供 | 自分で構築 |
| **メッセージチャネル** | プラットフォーム内蔵 | Hermes または OpenClaw（Telegram / WeChat / Feishu / DingTalk / ...） | 各自で統合 |
| **サービスロジック** | ビジュアル DAG ワークフロー | SKILL.md + プロンプト | コード定義 |
| **ナレッジベース** | 内蔵 RAG | MCP / ローカルドキュメント | 自分で構築 |
| **セッション記憶** | プラットフォームホスト | Hermes ネイティブ | 自分で管理 |
| **24/7 稼働** | プラットフォーム保証 | 自前サーバーにデプロイ | 自分で運用 |
| **データ管理** | プラットフォームサーバー | 自分のマシン | 自分のマシン |
| **学習曲線** | 低（ドラッグ＆ドロップ） | 中（Skill 作成 + 設定） | 高（フルスタック開発） |
| **柔軟性の上限** | プラットフォームの境界 | Skill + MCP の境界 | 無制限 |

## どれを選ぶべきか

**最速でプロトタイプを動かしたい** → Coze / Dify。ゼロコード、内蔵ナレッジベースとプラグイン、すぐに Feishu や WeChat に公開可能。トレードオフは、データがプラットフォーム側にあり、機能の上限がプラットフォームのロードマップに縛られること。限られた時間で成果を見たい非開発者に最適です。

**完全に制御可能で長期間稼働するカスタマーサービス Agent が欲しい** → Hermes（または OpenClaw）+ カスタマーサービス Skill。あなたはすでに Agent プラットフォームの全機能——メッセージチャネル、セッション記憶、MCP、Cron——を備えています。足りないのはカスタマーサービス領域の Skill ファイルだけです。この Skill に明記すべきこと：分類ルール（返品・交換 vs 注文照会 vs クレームエスカレーション）、ナレッジベース検索戦略、チケットツール呼び出し条件、人間への引き継ぎトリガー。自分のサーバーにデプロイして 24/7 稼働させます。

**自社製品に深く組み込み、カスタム動作が必要** → LangGraph。Agent ループのあらゆる詳細を制御できます。代償として、セッション管理、チャネルアダプター、長期記憶戦略、監視、運用を自分で書く必要があります。ただし `while True: response = LLM(...)` から始めてはいけません——LangGraph を使う場合でも、組み込みの永続化、状態管理、Human-in-the-loop プリミティブを活用します。

## 結論

> Hermes（または OpenClaw）にカスタマーサービス Skill を追加することは「妥協」ではありません。それ**こそが**カスタマーサービス Agent を構築することなのです。Skill は Agent の「頭脳」であり、Hermes と OpenClaw は「身体」——チャネルとランタイムはすでにそこにあります。あなたは頭脳を書くだけで、身体はすでにあります。Coze は頭脳と身体の両方をホスティングプラットフォーム上に提供します。スクラッチ開発は両方を自分で作ることを意味します。チャネル要件に合ったプラットフォームを選び、その上に Skill を追加してください。
